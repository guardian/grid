## 1. Architecture Overview

Kahuna communicates with Grid microservices **exclusively through a HATEOAS REST API** using the [Theseus](https://github.com/argo-rest/theseus) hypermedia client. The **single entry point** is the `media-api` root resource. All other service endpoints (cropper, loader, edits, collections, leases) are discovered by following link relations from that root. This means:

- **No hardcoded service URLs** exist in frontend code (except the media-api root injected via `<link rel="media-api-uri">` in the HTML page).
- Every API call goes through `mediaApi.root.follow('<rel-name>')` which resolves the correct URL at runtime.
- Authentication is cookie-based (pan-domain auth / OIDC). The Theseus HTTP client passes credentials via `withCredentials: true`.
- The Scala backend (`KahunaController`) only serves the SPA shell page and injects configuration (`_clientConfig`) including: `mediaApiUri`, `authUri`, feature flags, metadata templates, announcements, field aliases, etc.

### 2.1 media-api

**Service role**: Core image search, retrieval, deletion, permissions, session, syndication.

**Frontend module**: `services/api/media-api.js` (factory `mediaApi`)

**Entry point**: `client.resource(mediaApiUri)` → root resource

| Operation | Theseus Call | HTTP Method | Description | Used By (Kahuna UI) |
|---|---|---|---|---|
| **Search images** | `root.follow('search', {q, since, until, ...}).get()` | GET | Full-text + faceted search with ~20 params (q, ids, since, until, takenSince, takenUntil, modifiedSince, modifiedUntil, archived, valid, free, payType, uploadedBy, offset, length, orderBy, hasRightsAcquired, hasExports, syndicationStatus, countAll, persisted, useAISearch) | Search results page, polling for new images, recent uploads, "no taken date" count |
| **Find image by ID** | `root.follow('image', {id}).get()` | GET | Retrieve single image resource | Image detail page, polling after edits, delete polling |
| **Delete image** | `image.perform('delete')` | DELETE | Soft-delete an image | Delete image button (single + batch) |
| **Undelete image** | `root.follow('undelete', {id}).put()` | PUT | Restore a soft-deleted image | Undelete button |
| **Get session** | `root.follow('session').getData()` | GET | Current user info + permissions (`user.email`, `user.permissions.showPaid`) | User display, "my uploads" filter, default non-free filter, permissions-based UI |
| **Metadata search** | `root.follow('metadata-search', {field, q}).get()` | GET | Autocomplete for metadata fields (e.g. photoshoot titles) | Photoshoot typeahead |
| **Label search** | `root.follow('label-search', {q}).get()` | GET | Search existing labels | Label autocomplete |
| **Labels suggest** | `root.follow('suggested-labels', {q}).get()` | GET | Suggest labels for given query | Label suggestions |
| **Can user upload?** | `root.getLink('loader')` | HEAD-like | Check if 'loader' link exists → user has upload permission | Upload page guard, search query bar upload button |
| **Can user archive?** | `root.getLink('archive')` | HEAD-like | Check if 'archive' link exists | Archive button visibility |
| **Syndicate image** | `root.follow('syndicate-image', {id, partnerName, startPending}).post()` | POST | Send image to syndication partner (e.g. "Capture") | "Send to Photo Sales" button |
| **Get root config** | `root.get()` | GET | Fetch root resource with config + links; also used for auth check (401 → redirect) | App bootstrap, event `events:config-loaded` |
| **Image `getAction('delete')`** | Inspects HATEOAS actions on image | — | Check if delete action is available for current user | Deletability checks |
| **Image `getLink('edits')`** | Inspects HATEOAS links on image | — | Check if user can edit this image | Edit permission checks |
| **Image `getLink('crops')`** | Inspects HATEOAS links on image | — | Check if image can be cropped | Crop button visibility |
| **Image `getLink('download')`** | Inspects HATEOAS links on image | — | Get download URI for original | Download button / zip download |

#### Data Model Accessors (read from image resource, no extra HTTP calls)

Kahuna reads the following fields from the image resource returned by search/find. Kupua needs to handle all of them:

- `image.data.id`, `.cost`, `.valid`, `.invalidReasons`, `.persisted` (`.value`, `.reasons`)
- `image.data.metadata` (title, description, byline, credit, keywords, peopleInImage, dateTaken, location, specialInstructions, domainMetadata, etc.)
- `image.data.usageRights` (category, restrictions, etc.)
- `image.data.userMetadata.data` → `.metadata`, `.usageRights`, `.labels`, `.archived`, `.photoshoot`
- `image.data.source` (dimensions, orientedDimensions, mimeType, secureUrl, file)
- `image.data.thumbnail`
- `image.data.uploadInfo` (filename)
- `image.data.uploadedBy`, `.uploadTime`
- `image.data.exports` (crop list)
- `image.data.usages.data` (usage list with platform, status, references, metadata)
- `image.data.leases.data` (leases with access, startDate, endDate, active, leasedBy, createdAt, notes, lastModified)
- `image.data.collections` (collection list with pathId, path, cssColour)
- `image.data.syndicationStatus`, `.syndicationRights`
- `image.data.softDeletedMetadata` (indicates soft-deleted)
- `image.data.isPotentiallyGraphic` (for blurring)
- `image.data.optimisedPng` (for PNG-specific optimised view)
- `image.links` (various HATEOAS links including `ui:image`, `download`, `crops`, `optimised`, `optimisedPng`, `downloadOptimised`, `crop-download-*`)

### 2.2 metadata-editor (edits-api)

**Service role**: Edit image metadata, usage rights, labels, archive status, photoshoot.

**Frontend modules**: `services/api/edits-api.js` (factory `editsApi`), `edits/service.js` (factory `editsService`)

**Entry point**: `mediaApi.root.follow('edits')` → metadata-editor root

| Operation | Theseus Call | HTTP Method | Description | Used By |
|---|---|---|---|---|
| **Get usage rights categories** | `editsRoot.follow('usage-rights-list').getData()` | GET | Full list of all usage rights categories with properties, leases, restrictions | Usage rights editor, category dropdown |
| **Get filtered usage rights categories** | `editsRoot.follow('filtered-usage-rights-list').getData()` | GET | Filtered/preferred subset of categories | Usage rights editor (default view) |
| **Update metadata** | `image.data.userMetadata.data.metadata.put({data})` | PUT | Overwrite metadata fields (description, byline, credit, keywords, peopleInImage, location, domainMetadata, etc.) | Single field edit, batch edit, metadata templates |
| **Set metadata from usage rights** | `metadata.perform('set-from-usage-rights')` | POST | Auto-populate metadata fields based on the current usage rights category | After saving usage rights |
| **Update usage rights** | `image.data.userMetadata.data.usageRights.put({data})` | PUT | Set usage rights category + properties | Usage rights editor, batch apply, metadata templates |
| **Add label** | `image.data.userMetadata.data.labels.post({data: [labels]})` | POST | Add one or more labels to an image | Label editor, preset labels on upload |
| **Remove label** | `label.delete()` | DELETE | Remove a specific label resource | Label removal |
| **Set archived** | `image.data.userMetadata.data.archived.put({data: true/false})` | PUT | Archive or unarchive an image | Archive/unarchive buttons |
| **Add photoshoot** | `image.data.userMetadata.data.photoshoot.put({data: {title}})` | PUT | Set photoshoot title | Photoshoot editor |
| **Remove photoshoot** | `image.data.userMetadata.data.photoshoot.delete()` | DELETE | Clear photoshoot from image | Photoshoot removal |

#### Synchronisation Pattern (Critical for Kupua)

After every edit, Kahuna **polls the image resource** until the change is reflected in the API response (Thrall reindexes ES asynchronously). This is the `apiPoll` / `getSynced` pattern:

1. Perform edit (PUT/POST/DELETE to metadata-editor)
2. Receive updated edit resource
3. Poll `image.get()` repeatedly
4. Compare polled image's field with expected value
5. On match → emit `images-updated` event to refresh all UI components
6. On timeout → show error

This pattern is used for: metadata edits, usage rights, labels, archive status, photoshoot, collections, leases, crops.

**Batch operations** use `trackAll()` which processes images sequentially with progress events (`events:batch-operations:start`, `events:batch-operations:progress`, `events:batch-operations:complete`).

**Service role**: Collection tree CRUD, image ↔ collection associations.

**Frontend module**: `services/api/collections-api.js` (factory `collections`)

| **Delete collection** | `collection.perform('delete')` | DELETE | Remove a collection node | Collection panel delete button |
| **Check if deletable** | `node.getAction('remove')` | — | Check HATEOAS action availability | Collection node delete visibility |

**Service role**: Access-control leases (allow/deny use or syndication, with date ranges).

**Frontend module**: `services/api/leases.js` (factory `leaseService`)

**Entry point**: `mediaApi.root.follow('leases')` → leases service root

| Operation | Theseus Call | HTTP Method | Description | Used By |
|---|---|---|---|---|
| **Add lease** | `image.perform('add-lease', {body: newLease})` | POST | Add a single lease to an image | Lease editor save, batch add |
| **Add multiple leases** | `image.perform('add-leases', {body: leases})` | POST | Add multiple leases at once | Metadata template apply |
| **Replace leases** | `image.perform('replace-leases', {body: updatedLeases})` | PUT | Replace all leases on an image | Usage rights category change, batch lease replace |
| **Delete leases** | `image.perform('delete-leases')` | DELETE | Clear all leases from an image | Lease clear button |
| **Delete specific lease** | `leasesRoot.follow('leases', {id}).delete()` | DELETE | Delete a single lease by UUID | Individual lease deletion, rights-category lease removal |
| **Get leases by media ID** | `leasesRoot.follow('by-media-id', {id}).get()` | GET | Fetch current leases for an image | Lease display refresh |

#### Lease Types

- `allow-use` / `deny-use` — Use permissions with start/end dates
- `allow-syndication` / `deny-syndication` — Syndication permissions

#### Usage Rights → Lease Auto-Management

When usage rights are changed, Kahuna automatically:
1. Reads `category.leases` from the usage rights category definition
2. Creates leases based on the category's lease templates (`createCategoryLeases`)
3. Removes leases from the *previous* category (`removeCategoryLeases`)
4. This is triggered via `$rootScope.$broadcast('events:rights-category:add-leases')` / `events:rights-category:delete-leases`

After any lease mutation, Kahuna polls until `leases.lastModified` changes.

### 2.7 image-loader (loader)

**Service role**: Upload images to Grid.

**Frontend module**: `services/api/loader.js` (factory `loaderApi`), `upload/manager.js` (factory `uploadManager`, `fileUploader`)

**Entry point**: `mediaApi.root.follow('loader')` → loader root

| Operation | Theseus Call | HTTP Method | Description | Used By |
|---|---|---|---|---|
| **Prepare upload** | `loaderRoot.follow('prepare').post(mediaIdToFilenameMap)` | POST | Get pre-signed S3 PUT URLs for direct-to-bucket upload | Upload manager (when `shouldUploadStraightToBucket`) |
| **Upload file (legacy)** | `loaderRoot.follow('load', uploadInfo).post(imageData, {headers, transformRequest})` | POST | Upload binary image data through the loader service | Upload manager (fallback path) |
| **Import URI** | `loaderRoot.follow('import', {uri, identifiers}).post()` | POST | Import image from external URL | Drag-and-drop URL, Witness import |
| **Get upload statuses** | `loaderRoot.follow('uploadStatuses').get()` | GET | Get upload statuses for current user | "My uploads" section |
| **Get upload status** | `loaderRoot.follow('uploadStatus', {id}).get()` | GET | Poll single upload status by mediaId | Upload progress tracking |
| **Mark as queued** | `loaderRoot.follow('uploadStatus', {id}).post({status: 'QUEUED'})` | POST | Mark upload as queued after S3 PUT | Direct-to-bucket upload flow |

#### Upload Flow (Two Paths)

**Path A — Direct to S3** (when `shouldUploadStraightToBucket` is true):
1. Hash file with SHA-1 → `mediaId`
2. Call `prepare({mediaId: filename, ...})` → get pre-signed S3 PUT URLs
3. `fetch(preSignedUrl, {method: 'PUT', body: file, headers: {'x-amz-meta-media-id': mediaId}})` (direct browser → S3)
4. Call `markAsQueued(mediaId)` to notify the loader
5. Poll `uploadStatus` until `COMPLETED` or `FAILED`
6. On `COMPLETED`, poll the upload status resource to get the full image

**Path B — Through Loader** (legacy):
1. POST binary data to `loaderRoot.follow('load', {filename}).post(imageData)`
2. Receive image resource
3. Poll until indexed

#### Post-Upload Actions
- Auto-set description from filename if missing
- Apply preset labels (from localStorage)
- Both via editsService calls

### 2.8 auth

**Service role**: OIDC / pan-domain authentication.

**Frontend access**: Cookie-based sessions managed by [pandular](https://github.com/guardian/pandular) (AngularJS module).

| Mechanism | Description | Used By |
|---|---|---|
| **Login redirect** | On 401/419 from media-api root, redirect to `authUri + '/login?redirectUri=...'` | App bootstrap |
| **Session re-establishment** | Pandular intercepts 401 responses and tries to re-establish the session via an iframe at `pandular.reAuthUri` (timeout: 7000ms) | Global HTTP interceptor |
| **Auth failure** | If re-establishment fails, emit `pandular:re-establishment:fail` → show "auth failed" banner | Global error handler |
| **Cookie-based auth** | All HTTP requests include cookies (`withCredentials: true`) | Every API call |

## 3. HATEOAS / Theseus Client Pattern

All API communication uses the Theseus hypermedia client which:

1. **Discovers** endpoints by following link relations from the root
2. **Caches** link traversals (memoized roots for cropper, loader, leases, edits, collections)
3. **Performs actions** by name (e.g. `image.perform('delete')`, `image.perform('add-lease', {body})`)
4. **Checks capabilities** via `getAction('name')` and `getLink('rel')` — returns undefined if not available

**Kupua implication**: The new React frontend must either:
- Use the same Theseus client (or a modern equivalent)
- Or build a typed API client that follows the same link-relation discovery pattern
- All permission checks are HATEOAS-driven (no hardcoded role checks in the frontend)

## 5. Cross-Cutting Concerns

### 5.1 Optimistic Polling (`apiPoll`)

Used after every write operation. Pattern:
```
mutation → poll image.get() → compare → resolve or retry (with backoff)
```

Kupua should consider: WebSocket/SSE for real-time updates, or replicate the polling pattern.

### 5.3 iframe Embedding / postMessage

Kahuna can be embedded as an iframe (e.g. by Composer). It posts messages to the parent window on:
- `events:crop-selected` → `{image: {uri, data}, crop: {data}}`
- `events:crop-created` → `{image: {uri, data}, crop: {uri, data}}`
- `events:crops-created` → `{images: [{image, crop}, ...]}`

MIME types for drag-and-drop interop:
- `application/vnd.mediaservice.image+json`
- `application/vnd.mediaservice.images+json`
- `application/vnd.mediaservice.crops+json`
- `application/vnd.mediaservice.kahuna.uri`
- `application/vnd.asset-handle+json`

### 5.4 Drag and Drop

Images/crops can be dragged from Kahuna into:
- Collection tree nodes (adds image to collection)
- External applications (provides image/crop data in multiple MIME types)

### 5.5 Quotas Page

Standalone page (`/quotas`) that directly fetches `{mediaApiUri}/usage/quotas` via `fetch()` with credentials. Displays supplier usage vs. quota limits.

### 5.6 Graphic Content Blurring

Client-side feature that blurs potentially graphic images based on:
- Server flag `image.data.isPotentiallyGraphic`
- Client-side keyword/description matching
- Controlled by cookie `SHOULD_BLUR_GRAPHIC_IMAGES`

## Summary for Kupua Developers

### Must-Have API Integrations

1. **media-api** — The foundation. Search, image retrieval, session, delete, undelete, syndicate. Everything starts from the root resource.
2. **metadata-editor** — All editing flows (metadata fields, usage rights, labels, archive, photoshoot). Includes the async polling pattern.
3. **collections** — Full tree CRUD + image association. Drag-and-drop support.
4. **cropper** — Crop creation (sized + full), crop retrieval, crop deletion, crop download links.
5. **leases** — CRUD for access-control leases, auto-management from usage rights categories.
6. **image-loader** — Two upload paths (direct-to-S3, legacy through loader), URI import, upload status tracking.

### Key Architectural Patterns to Replicate

1. **HATEOAS link following** — All URLs are discovered, not hardcoded.
2. **Optimistic polling after writes** — Every mutation is followed by polling until the change is reflected.
3. **Batch operation tracking** — Sequential processing with progress events.
4. **iframe postMessage API** — For embedding in Composer and other tools.
5. **Drag-and-drop MIME types** — Custom vendor MIME types for cross-app integration.
6. **Server-driven permissions** — All capability checks use HATEOAS actions/links, not client-side role logic.
