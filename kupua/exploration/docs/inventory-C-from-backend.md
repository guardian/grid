## Out-of-scope appendix

≤20 items, one line each.

1. **`GET /management/whoAmI`** — inner-service status check across service mesh depth; not a user-facing capability.
2. **`GET /management/manifest`** — returns build manifest (version, git sha); ops tooling.
3. **`customSpecialInstructions` and `customUsageRestrictions` are Play config maps** — configured in `application.conf` per environment, not via any API; kupua would need to fetch these from media-api root or hardcode them.
4. **`fuzzySearchEnabled` / `fuzzySearchEditDistance` / `fuzzyMaxExpansions` / `fuzzySearchPrefixLength`** — server-side search behaviour config flags; kahuna has no control over them; they affect all `q=` free-text queries silently.
5. **`useRuntimeFieldsToFixSyndicationReviewQueueQuery`** — server config flag that injects an ES runtime field mapping when querying `AwaitingReviewForSyndication`; transparent to clients.
6. **Embedding lambda uses `global.cohere.embed-v4:0` on Bedrock with `output_dimension: 1536`** — but ES mapping stores only 256-dim (`cohereEmbedV4.image`). The lambda likely truncates/projects. Discrepancy worth noting for AI search accuracy analysis.
7. **`aiSearchEmbeddingCacheMaxSize`** — server-side in-memory cache for query embeddings (Scaffeine); reduces Bedrock calls for repeated queries within the same server process.
8. **`shouldDisplayOrgOwnedCountAndFilterCheckbox` + `maybeAgencyPickQuery`** — config flags that control whether the ticker aggregations (org-owned count, agency-pick count) appear in search responses. If not configured, `extraCounts` returns empty maps.
9. **`agencyPicksColour`** — server config string; controls the `backgroundColour` on the agency-picks ticker count badge. Not relevant to kupua currently.
10. **Thrall `listImageIdsWithPrefix` + `listImageIdsWithUnexpectedFormat`** — internal Thrall admin queries for ID migration tooling; not accessible via any media-api route.
11. **`addCollection` action has no write-permission check.** Confirmed in A's out-of-scope appendix item 18; backend `imageActions` confirms: `addCollectionAction -> true` (always included for Internal tier). No `withWritePermission` gate.
12. **Thrall migration index** — all write operations fan out to both `imagesCurrentAlias` and `imagesMigrationAlias` when a migration is in progress; clients never see this — it's transparent.
13. **`sourceAsImageWrapper`** — Thrall exposes a `getImageVersion` endpoint for optimistic-lock checks during updates; not a media-api route.
14. **`SearchQueryTimeout = 10s`** — hard ES query timeout on all search requests; a partial result may be returned if ES times out (because `allow_partial_search_results` defaults to true in the cluster config).
15. **`collections.pathHierarchy` field uses a custom hierarchy-analysed field type** — hierarchy analysis tokenises paths on `/` separator for prefix-matching; `~"path"` CQL queries against this field work natively in ES without additional post-processing.
16. **`usages[].references[]` nested object** — usage references have a `{type, uri, name}` structure (e.g. Composer article reference); not surfaced in kupua's current usage display.
17. **`fileMetadata` is lazy by default** — `GET /images/:id` returns `fileMetadata` as a stub (URI only, no content) unless `include=fileMetadata` query param is set; `GET /images/:id/fileMetadata` is the dedicated full-metadata route.
18. **`recordDownloadAsUsage` download tracking** — when `config.recordDownloadAsUsage` is true, `downloadOriginalImage` and `downloadOptimisedImage` POST to the usage service automatically; this is a server-side side-effect of the download routes that neither A nor B fully documents.
19. **Usage email-parsing via JavaMail (`UsageStore.extractEmail`)** — supplier usage quotas are sourced from an email (base64 CSV attachment) parsed server-side; not a webhook or API; S3-bucket-triggered.
20. **Thrall's `getInferredSyndicationRightsImages` and `getLatestSyndicationRights`** — internal Thrall queries used to propagate syndication rights across photoshoot members; not externally accessible but explain why `syndicationRights.isInferred` exists in ES.
